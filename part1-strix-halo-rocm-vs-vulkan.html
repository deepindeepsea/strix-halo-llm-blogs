A field series · Results · Part 1

ROCm or Vulkan on Strix Halo? We benchmarked both, head to head, and the iGPU picked a side.

The first measured chapter of the R9700 method applied to an APU. Two complete llama.cpp backends — ROCm/HIP and Vulkan/RADV — built from the same source commit, run on the same Radeon 8060S iGPU, across four models from 8B to 70B, single-user and batched. One backend wins the desktop. The other wins the server. Here is exactly where the line is, and how to reproduce every number.

AMD Field AI Enthusiast · by Pradeep Nallimelli · June 2026 · status: RESULTS — measured on real hardware

The one-paragraph verdict

On the Ryzen AI Max+ 395’s integrated Radeon 8060S, Vulkan/RADV is the better single-user backend — decisively so on a Mixture-of-Experts model (+37% decode on Qwen3-30B-A3B: 96.4 vs 70.4 tok/s), and modestly (2–3%) on dense models. But there is a crossover: under heavy batching, ROCm scales further and overtakes Vulkan at 16 concurrent streams (272 vs 180 tok/s aggregate decode, +51% for ROCm). The practical rule: Vulkan for interactive and small-batch; ROCm for many-stream batch serving. Same llama.cpp build (063d9c1), no per-backend cherry-picking.

The bench · one APU, two backends, same commit

The whole point of a head-to-head is that nothing differs except the backend. Both binaries were compiled from the same llama.cpp checkout on the same machine; only the GPU backend flag changed. Then every model was run through an identical sweep.

Build both backends from one source tree (the exact commands):

git clone https://github.com/ggml-org/llama.cpp && cd llama.cpp

# Backend A — ROCm / HIP
cmake -B build -DGGML_HIP=ON -DAMDGPU_TARGETS=gfx1151 \
      -DGGML_HIP_ROCWMMA_FATTN=ON -DGGML_HIP_NO_VMM=ON \
      -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release -j $(nproc)

# Backend B — Vulkan / RADV
sudo apt install -y libvulkan-dev glslc glslang-tools glslang-dev \
                    spirv-headers spirv-tools mesa-vulkan-drivers
cmake -B build-vulkan -DGGML_VULKAN=ON -DLLAMA_CURL=OFF \
      -DCMAKE_BUILD_TYPE=Release
cmake --build build-vulkan --config Release -j $(nproc)

The Vulkan toolchain needs the SPIR-V headers (spirv-headers + glslang-dev) or the CMake config step fails looking for SPIRV-HeadersConfig.cmake. That single apt line is the only gotcha.

The pattern is unambiguous: Vulkan wins single-user decode on every model. On dense models the gap is small (the kernels are bandwidth-bound and both backends saturate the same ~256 GB/s wall). On the MoE model the gap explodes to +37% — RADV’s expert-routing / gather path on gfx1151 is simply more efficient than the current ROCm path. That MoE advantage is the single most important finding for this APU, because MoE is exactly the model shape that makes a bandwidth-limited chip feel fast.

Result 2 · flash attention behaves oppositely on the two backends

A tuning trap worth pinning down: the “always use -fa 1” folklore is backend-specific on this iGPU.

BackendFA off · prefillFA on · prefillFA off · decodeFA on · decodeBest setting
ROCm1067.1828.566.270.4FA off for prefill, on for decode
Vulkan1070.51174.693.996.4FA on — wins both

On Vulkan, flash attention is a free win: it improves both prefill and decode, so leave it on. On ROCm it’s a trade: FA on costs you ~22% prefill throughput on the MoE while buying ~6% decode. If you serve long prompts on ROCm, FA off may actually be the right call — the opposite of the usual advice.

Result 3 · prefill at depth — where Vulkan pulls away

Short-prompt prefill is close, but as the prompt grows, the backends diverge hard. Qwen3-30B-A3B, FA on, prefill tok/s at increasing prompt length:

Prompt lengthROCmVulkanVulkan advantage
512 tokens7701159+51%
2048 tokens3091106+258%
4096 tokens211966+358%

This is dramatic. ROCm prefill collapses with prompt depth (770 → 211 tok/s), while Vulkan stays remarkably flat (1159 → 966). At a 4K prompt, Vulkan prefills 4.6× faster. For any retrieval, long-context, or document-heavy workload on this APU, Vulkan isn’t just a little better — it’s a different class of usable.

Result 4 · the micro-batch knob (-ub)

The cheapest tuning lever is -ub (physical micro-batch). Qwen3-30B-A3B prefill, FA on, sweeping -ub 256/512/1024:

-ubROCm pp512Vulkan pp512
256705868
512 (default)8331152
10247911154

Takeaway: the default -ub 512 is already near-optimal. ROCm peaks exactly at 512; Vulkan is flat from 512–1024 and you can push to 1024 if you have the memory headroom. Going below 512 hurts both. No exotic tuning required.

Result 5 · the capacity flex — a 70B on an iGPU, no reboot

This is the move a discrete 32 GB card cannot make. Llama-70B Q4_K_M is 39.6 GiB — it loaded fully onto the integrated GPU at -ngl 99 through the 96 GiB GTT pool, with no BIOS change and no reboot.

ROCm · 70B Q4
112.6 tok/s prefill · 5.01 tok/s decode
Loads and runs entirely on the iGPU. Decode is bandwidth-bound: ~40 GB streamed per token against ~256 GB/s sets the ~5 tok/s ceiling.
Vulkan · 70B Q4
114.3 tok/s prefill · 5.15 tok/s decode
A hair ahead again. At this size the two backends converge — both are simply waiting on memory, and the silicon, not the driver, sets the pace.

Five tokens per second isn’t interactive, but it is real, local, single-node 70B inference on a fanless-class APU — useful for batch jobs, overnight document processing, and privacy-bound workloads where capacity matters more than latency. The lesson echoes the R9700 series: on bandwidth-limited silicon, pick MoE over dense when you can. A 30B MoE gives you ~96 tok/s; a 70B dense gives you ~5.

Result 6 · the crossover — concurrency flips the winner

Single-user is only half the story. Driving Qwen3-30B-A3B with llama-batched-bench at 1–16 parallel sequences (512-token prompt, 128-token gen each) reveals the most important operational nuance of the whole campaign: the backends trade places.

Concurrent streamsROCm agg. decodeVulkan agg. decodeLeader
170.091.9Vulkan
298.6130.8Vulkan
4133.6177.6Vulkan
8182.1199.8Vulkan
16272.3180.2ROCm +51%

Vulkan leads from 1 to 8 streams. But at 16 concurrent streams Vulkan plateaus (~180 tok/s) while ROCm keeps climbing to 272 tok/s — a 51% reversal. ROCm’s scheduler and HIP kernels scale batched decode better once the GPU is saturated with parallel work; Vulkan’s decode throughput actually regresses past 8. If you are building a multi-user or agentic fleet on this APU, ROCm is the backend that gives you headroom.

The decision, stated plainly: the right backend depends on how many people (or agents) you serve at once. There is no single winner — there is a crossover at roughly 8–16 concurrent streams.

Reproduce every number

Nothing here is hand-waved. The exact sweep commands, runnable on any Strix Halo box once both backends are built:

# Single-user matrix (per model) — swap build/bin for build-vulkan/bin to switch backend
BIN=llama.cpp/build/bin      # ROCm   (prefix env below)
# BIN=llama.cpp/build-vulkan/bin   # Vulkan (no env needed)
ENV="HSA_OVERRIDE_GFX_VERSION=11.5.1 LD_LIBRARY_PATH=$BIN:/opt/rocm/lib"   # ROCm only

# Result 1 + 2 — FA sweep (prefill + decode)
$ENV $BIN/llama-bench -m $MODEL -ngl 99 -fa 0,1 -p 512 -n 128 -r 3
# Result 3 — prefill depth
$ENV $BIN/llama-bench -m $MODEL -ngl 99 -fa 1 -p 512,2048,4096 -n 0 -r 2
# Result 4 — micro-batch sweep
$ENV $BIN/llama-bench -m $MODEL -ngl 99 -fa 1 -b 2048 -ub 256,512,1024 -p 512 -n 0 -r 2

# Result 6 — concurrency / batched throughput (note: newer llama.cpp needs '-fa on')
$ENV $BIN/llama-batched-bench -m $MODEL -c 12288 -b 2048 -ub 512 \
     -npp 512 -ntg 128 -npl 1,2,4,8,16 -ngl 99 -fa on

Models (Q4_K_M GGUF) pulled from HuggingFace: Qwen3-30B-A3B, Qwen3-14B, Llama-3.1-8B-Instruct, and a 70B Q4. Build commit 063d9c1; ROCm 7.2.0; Mesa RADV 25.2.8; kernel 6.17-oem.
